System design

Current architecture (coherent with v1.0.1 workflow)

3DS plugin

Sends UDP events with title ID and user token.

UDP :5005 schemaVersion=1
Bridge server

Routes events by token, serves API/WS, and stores tokens in SQLite.

HTTP :8766 WS :8765 DRPC_DB_PATH
Tauri app

Installs token/plugin, connects to WS, and updates Discord RPC locally.

Install flow Autostart Rich Presence
Isolation model: one token = one user path (plugin build + WS auth + event routing).

Install workflow

Provisioning and runtime flow

Desktop app install
  -> POST /token
  <- { token, udp_host, udp_port }
  -> GET /plugin/build?token=<tok>
  <- default.3gx
  -> copy plugin to SD:/luma/plugins/default.3gx (or per-title)
  -> app daemon connects to WebSocket with {"type":"auth","token":"..."}

Token management

Tokens are now persisted server-side in SQLite to survive restart and avoid duplicate generation issues.

Plugin artifact naming

Client and docs are aligned on default.3gx for consistency with the current deployment pipeline.

Client resilience

The desktop app retries transient 503 responses during plugin build download to reduce failed installs.
